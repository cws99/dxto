# CLAUDE.md — DxTO (Diagnostic Test Ontology)

Instructions for Claude Code when working in this repository.

## What this repo is

An OBO-style ontology of diagnostic tests, generated by the Ontology
Development Kit (ODK), structured to mirror monarch-initiative/mondo. Tests
are linked to MONDO disease classes via an `is_diagnostic_for` relation.
Executable clinical decision logic (CQL/FHIR) lives in a SEPARATE repo
(`dxto-cds`) and must NOT be added here.

## Hard rules

- The file to edit is `src/ontology/dxto-edit.obo`. Never edit generated
  release files (`dxto.obo`, `dxto.owl`, `dxto.json`).
- NEVER hand-edit `src/ontology/Makefile` — it is ODK-generated. To change
  the build, edit `src/ontology/dxto-odk.yaml` and re-run the ODK update.
- Mint IDs as `DxTO:nnnnnnn`, zero-padded to 7 digits, sequential. Do not
  reuse retired IDs.
- LOINC and SNOMED are NOT imported as OBO ontologies. Their links are
  `skos:exactMatch` mappings managed as SSSOM TSVs in `components/`.

## ID & mapping conventions

- Lab tests anchor to LOINC; procedures anchor to SNOMED CT (matching OMOP's
  standard-concept choice). Record the OMOP domain (Measurement / Procedure /
  Observation) as an annotation.
- Use `skos:exactMatch` only for true 1:1 equivalence; use a looser
  cross-reference predicate otherwise. Preserve this distinction — it is the
  core value of the MONDO approach.

## Relation layer

- `components/dxto-diagnostic-relations.tsv` columns:
  `disease | test | role | tier | evidence | guideline_provenance`.
- `role` ∈ {confirmatory, screening, supportive, exclusionary}.
- Templated into OWL via the DOSDP pattern in `src/patterns/`.

## Before opening a PR

Run `cd src/ontology && sh run.sh make test` and make sure QC passes.
